When invoked with /commit, follow this flow:

## Step 1: Run Tests
- Run: pytest
- If any fail, stop and report failures in detail
- Do NOT proceed if tests fail

## Step 2: Review Changes
- Run: git diff --staged (to see staged changes)
- Run: git status (to see working tree)
- Understand what changed and why

## Step 3: Determine Type and Scope
From the actual changes, derive:
- Type: feat | fix | refactor | test | docs | chore | style | perf | ci
- Scope: api | db | auth | scheduler | services | build (pick most relevant)

Examples:
- Adding an endpoint → feat(api)
- Fixing an auth bug → fix(auth)
- Refactoring the service layer → refactor(services)
- Adding a model + migration → feat(db)
- Updating tests → test(api or db)
- Updating docs → docs
- Bumping a dependency → chore(build)

## Step 4: Write Commit
- Subject: <type>(<scope>): description
- Rules:
  - ≤72 characters
  - lowercase
  - no period at end
  - imperative mood ("add" not "added")
- Body: only if "why" is non-obvious
- Include Co-Authored-By footer

## Step 5: Execute
- Create and execute the commit
- Verify with git log

## Important
- MUST have all tests passing before commit
- MUST use imperative mood
- MUST match Conventional Commits format
