# ADR 0002: URL versioning strategy and OpenAPI contract enforcement

**Status:** Accepted  
**Date:** 2026-06-05  
**Issue:** #6

## Context

The frontend and backend were tightly coupled — the frontend called `/api/...`
paths that had no versioning, and the API was documented only in hand-written
Markdown files that drifted from the actual implementation. There was no
automated way to detect when a code change broke the API contract.

## Decision

1. **URL versioning:** All API routes are prefixed with `/v1/` (e.g.,
   `/api/v1/chores`). Status and metrics endpoints remain unversioned.
   The current version is recorded in `API_VERSION` at the repo root.

2. **OpenAPI golden snapshot:** `docs/api/openapi.json` is a committed
   snapshot generated by `scripts/generate_openapi.py`. It is the source
   of truth for the current API contract.

3. **CI enforcement:** A `api-contracts` CI job generates a fresh OpenAPI
   schema on every PR and runs `oasdiff breaking` against the committed
   snapshot. Breaking changes cause CI to fail unless `API_VERSION` has
   been incremented.

4. **Hand-written docs removed:** `docs/api/*.md` files are deleted.
   The FastAPI `/docs` UI (Swagger) and the committed `openapi.json` replace
   them as the authoritative API reference.

## Rationale

- URL versioning is simple, explicit, and visible in logs and URLs.
- Committing the OpenAPI snapshot makes breaking changes reviewable in PRs.
- `oasdiff` is a well-maintained, standard tool for OpenAPI diff/breaking
  change detection.
- Removing hand-written docs eliminates the drift problem at its source.

## Consequences

- All frontend API calls use `/api/v1/` prefix. The `VITE_API_URL`
  environment variable should point to the versioned base (`/api/v1`).
- To introduce a breaking API change: increment `API_VERSION`, mount new
  routes under `/api/v{N}/`, and regenerate `openapi.json`.
- Existing tokens become invalid when the JWT secret is rotated (acceptable,
  documented in deployment docs as a v2.0.0 breaking change).
